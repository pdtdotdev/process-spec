# Introduction to PROCESS.md

Standard Operating Procedures (SOPs) usually live in documents, wikis, tickets, or runbooks. They explain how work should happen, but they are not directly executable.

Teams often automate these processes by moving the workflow into long prompt templates, hardcoded scripts, or visual state machines. That makes the process harder for the operational owner to inspect, change, and audit.

`PROCESS.md` defines executable SOPs in prose-first Markdown. The same file can be read by humans, reviewed like documentation, and compiled by AI runtimes into bounded execution steps.

Tools, skills, schemas, and subprocesses are core reference types in the standard. Implementations can add extensions, but standard PROCESS.md behavior does not require a runtime declaration.

---

## Core Philosophy

`PROCESS.md` is built on four principles:

### 1. Separation of Concerns
Separate what an agent can do from what it should do next.

* **Skills (`SKILL.md`)** define reusable capabilities, such as reviewing SQL safety or formatting a CSV report.
* **Processes (`PROCESS.md`)** define context-specific workflows, including steps, approvals, dependencies, and expected outputs.

### 2. Prose-First Ownership
Workflows should stay close to the teams that run them. Markdown keeps the process readable, editable, version-controlled, and auditable without requiring a state-machine DSL.

### 3. Step Isolation
A runtime parses the workflow into ordered steps and executes one step at a time. The agent receives the current step, the global process context, and only the tools or references allowed for that step.

### 4. Execution Safety
Natural language is not a security boundary. Tool permissions, approval gates, write access, and dry-run behavior are enforced during execution.

---

## Why Standardize?

A shared specification makes operational workflows portable, inspectable, and enforceable:

* Share workflows across agents, frameworks, and orchestrators.
* Keep one source of truth for human operators and AI runtimes.
* Audit which instructions, tools, outputs, and approvals were used.
* Enforce approval and permission boundaries during execution.
