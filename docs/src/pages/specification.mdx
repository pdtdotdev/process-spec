# PROCESS.md Specification

A conforming `PROCESS.md` document is structured for compilation and step-by-step execution. This guide covers document layout, references, execution behavior, and validation.

---

## Document Structure

Every `PROCESS.md` file has three blocks in this order:

1. **YAML frontmatter**
2. **`# Description`**
3. **`# Workflow`**

```
+------------------------------------+
|                                    |
|  YAML Frontmatter                  |
|  ---                               |
|  id: growth_experiment_review      |
|  ...                               |
|  ---                               |
|                                    |
+------------------------------------+
|                                    |
|  # Description                     |
|  Global context and goals...       |
|                                    |
+------------------------------------+
|                                    |
|  # Workflow                        |
|                                    |
|  ## Step 1: Lookup Experiments     |
|  Instructions...                   |
|                                    |
|  ## Step 2: Analyze Results        |
|  Instructions...                   |
|                                    |
+------------------------------------+
```

### 1. YAML Frontmatter
The file starts with YAML frontmatter enclosed by `---`.

```yaml
---
id: growth_experiment_review
name: Growth Experiment Review and Analysis
version: 0.1.0
owner: growth-ops
status: active
description: Analyzes completed product growth experiments and updates downstream spreadsheets.
tags: [growth, analytics, operations]
---
```

* **`id`**: Unique, stable `snake_case` identifier.
* **`name`**: User-facing process name.
* **`version`**: Semantic version (e.g., `0.1.0`).
* **`owner`**: Department, role, or team responsible for the process.

### 2. Description
The `# Description` section follows the frontmatter. It defines process context, scope, constraints, and goals.

```markdown
# Description

This process governs the weekly ingestion and review of growth experiment results. 
It ensures all experiment metrics are cross-checked, formatted correctly, and published.

## Scope
- Covers product and marketing metrics.
- Excludes backend engineering experiment validations.
```

### 3. Workflow
The `# Workflow` section contains ordered execution steps. Each step uses a level-2 heading:

```markdown
## Step <N>: <Name>
```

```markdown
# Workflow

## Step 1: Retrieve Experiment Metadata
Use the `tool/experiment_lookup` tool to fetch metadata for the experiment ID.
If the metadata cannot be found, abort the process and notify the owner.

## Step 2: Analyze Metrics
Analyze the results using the `skill/experiment-analysis` capability.
Validate the formatted output payload against `schema/experiment-review`.
```

---

## Reference Resolution Schema

`PROCESS.md` uses inline code references to bind steps to standard resource types:

* **`` `skill/<id>` ``** references reusable guidance for the current step.
* **`` `tool/<id>` ``** references an executable tool or integration.
* **`` `schema/<id>` ``** references a structured output or payload contract.
* **`` `process/<id>` ``** references another process as a subprocess.

---

## Execution Model

During execution, an engine resolves references, loads the resources needed for the current step, enforces permissions, validates outputs, and records execution evidence.

Implementations must parse the core reference types and report unresolved references as execution errors. Specific runtimes can add resource types, policies, or integrations as extensions.

---

## Compiler Validation Checklist

Before execution, a compiler should validate:

1. **Frontmatter integrity:** Required keys (`id`, `name`, `version`, `owner`) are present and valid.
2. **Step sequence:** Steps start at `Step 1` and increment by 1 without gaps.
3. **Reference syntax:** Inline references use `` `type/id` `` syntax.
4. **Reference types:** Core reference types are `skill`, `tool`, `schema`, and `process`.
